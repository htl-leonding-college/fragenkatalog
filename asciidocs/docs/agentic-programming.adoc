== Einführung

Agentisches Programmieren (Agentic Coding) beschreibt einen Paradigmenwechsel in der Softwareentwicklung: Statt passiver Code-Vervollständigung arbeitet Claude Code als autonomer Agent, der eigenständig Code erkundet, editiert und erklärt – über das gesamte Projekt hinweg.

Dieser Leitfaden beschreibt einen strukturierten Workflow, bei dem die *fachliche Spezifikation* (WAS) klar von der *technischen Implementierung* (WIE) getrennt wird.

== Installation von Claude Code

=== macOS (mit Homebrew)

==== Variante 1: Native Installer (empfohlen)

Der native Installer benötigt kein Node.js und aktualisiert sich automatisch:

[source,bash]
----
curl -fsSL https://claude.ai/install.sh | bash
----

Nach der Installation ein neues Terminal-Fenster öffnen, dann:

[source,bash]
----
claude --version
----

==== Variante 2: Homebrew Cask

[source,bash]
----
brew install --cask claude-code
claude --version
----

IMPORTANT: Homebrew-Installationen aktualisieren sich *nicht* automatisch. Regelmäßig `brew upgrade claude-code` ausführen.

==== Variante 3: npm (falls Node.js bevorzugt wird)

Voraussetzung: Node.js 18+

[source,bash]
----
# Falls Node.js fehlt
brew install node

# Claude Code installieren
npm install -g @anthropic-ai/claude-code

# Bei "command not found":
echo 'export PATH="$(npm config get prefix)/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

claude --version
----

=== Linux

==== Variante 1: Native Installer (empfohlen)

[source,bash]
----
curl -fsSL https://claude.ai/install.sh | bash
----

Falls `command not found` erscheint:

[source,bash]
----
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
----

==== Variante 2: Homebrew (falls installiert)

[source,bash]
----
brew install --cask claude-code
----

==== Variante 3: npm

[source,bash]
----
# npm-Verzeichnis ohne sudo einrichten
mkdir ~/.npm-global
npm config set prefix '~/.npm-global'
echo 'export PATH=~/.npm-global/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

# Installation
npm install -g @anthropic-ai/claude-code
claude --version
----

CAUTION: Niemals `sudo npm install -g` verwenden – das verursacht Berechtigungsprobleme.

=== Windows

==== Variante 1: PowerShell (Native Installer)

[source,powershell]
----
irm https://claude.ai/install.ps1 | iex
claude --version
----

==== Variante 2: CMD

[source,cmd]
----
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd
----

==== Variante 3: WSL (Windows Subsystem for Linux)

WSL 1 und WSL 2 werden unterstützt. Nach WSL-Setup die Linux-Anweisungen befolgen.

==== Variante 4: Git Bash

Zuerst https://gitforwindows.org/[Git for Windows] installieren, dann:

[source,bash]
----
curl -fsSL https://claude.ai/install.sh | bash
----

=== Authentifizierung

Nach der Installation:

[source,bash]
----
claude auth login
----

Dies öffnet den Browser für OAuth-Authentifizierung. Alternativ:

[source,bash]
----
export ANTHROPIC_API_KEY="sk-ant-..."
----

NOTE: Claude Code erfordert einen Pro, Max, Teams, Enterprise oder Console Account.

=== Schritt 2: Claude Code starten und Fragen beantworten

Starte Claude Code im Projektverzeichnis:

[source,bash]
----
cd mein-projekt
claude
----

==== Initialer Prompt

[source]
----
Lies bitte specs/story.adoc und analysiere die Anforderungen.

Stelle mir gezielte Fragen, um:
1. Die fachlichen Anforderungen vollständig zu verstehen
2. Alle Unklarheiten zu beseitigen
3. Die technischen Entscheidungen zu validieren

Formatiere deine Fragen strukturiert nach Kategorien.
Frage NICHT nach Code oder Implementierungsdetails –
das kommt später.
----

==== Typische Fragen von Claude

Claude wird Fragen stellen wie:

.Fachliche Fragen
* Welche Benutzerrollen gibt es?
* Was passiert bei Stornierung einer Bestellung?
* Gibt es Rabattsysteme oder Gutscheine?
* Wie werden Retouren behandelt?

.Technische Fragen
* Soll die API versioniert werden?
* Welche Authentifizierung (JWT, OAuth2, ...)?
* Gibt es Performance-Anforderungen (Requests/Sekunde)?
* Soll Event Sourcing verwendet werden?

.Klärungsfragen
* Was genau bedeutet "Bestellstatus verfolgen"?
* Wer darf Bestellungen stornieren?
* Wie lange werden Daten aufbewahrt?

TIP: Beantworte die Fragen ausführlich. Je mehr Kontext Claude hat, desto besser werden die generierten Spezifikationen.

=== Schritt 3: FSD.adoc generieren lassen

Nach dem Frage-Antwort-Dialog:

[source]
----
Erstelle basierend auf unserer Diskussion eine vollständige
fachliche Spezifikation als specs/fsd.adoc.

Verwende folgende Struktur:
- Glossar mit Ubiquitous Language
- Akteure und ihre Berechtigungen
- Anwendungsfälle (Use Cases) mit Vor-/Nachbedingungen
- Geschäftsregeln mit IDs
- Fachliches Datenmodell (ohne technische Details)
- Akzeptanzkriterien

Die Spezifikation soll von Fachexperten lesbar sein –
keine technischen Implementierungsdetails.
----

== Architektur

=== Hexagonale Architektur

[source]
----
src/main/java/com/beispiel/projekt/
├── domain/           # Reine Fachlogik
│   ├── model/        # Entities, Value Objects
│   ├── service/      # Domain Services
│   └── port/
│       ├── in/       # Eingehende Ports (Use Cases)
│       └── out/      # Ausgehende Ports (Repositories)
├── application/      # Use-Case-Implementierungen
│   └── usecase/
└── adapter/          # Technische Adapter
├── in/
│   └── web/      # REST Controller
└── out/
└── persistence/  # JPA Repositories
----

=== Dependency Rule

[IMPORTANT]
====
* `domain/` kennt NUR sich selbst – keine Framework-Imports
* `application/` kennt `domain/`
* `adapter/` kennt `application/` und `domain/`
====

== Coding-Konventionen

=== Allgemein
* Package-Prefix: `com.beispiel.projekt`
* Keine Wildcards in Imports
* Records für Value Objects
* Optional statt null-Returns

=== Naming
* Klassen: PascalCase
* Methoden/Variablen: camelCase
* Konstanten: SCREAMING_SNAKE_CASE
* Packages: lowercase

=== Domain-Regeln
* Entities: Fachbegriffe aus FSD-Glossar verwenden
* Keine @Entity, @Id etc. in domain/model/
* Validierung in Konstruktoren/Factory-Methoden

== Mapping: FSD → Code

[cols="1,2,2"]
|===
| FSD-Element | Code-Artefakt | Beispiel

| Glossar-Begriff
| Klasse in `domain/model/`
| `Bestellung.java`

| Anwendungsfall
| Interface in `domain/port/in/`
| `BestellungAufgebenUseCase.java`

| Geschäftsregel
| Methode in Entity oder Domain Service
| `Bestellung.validiereMindestwert()`

| Akteur
| Security Role
| `ROLE_KUNDE`, `ROLE_SACHBEARBEITER`
|===

== Test-Strategie

[cols="1,2,2"]
|===
| Ebene | Scope | Technologie

| Unit Tests
| Domain-Logik

| Integration Tests
| Adapter, Repositories

== Workflow für Claude Code

=== Bei neuen Features

1. `specs/fsd.adoc` konsultieren
2. Relevanten Use Case identifizieren
3. Port-Interface in `domain/port/in/` erstellen
4. Domain-Klassen implementieren (falls nötig)
5. Use-Case-Implementierung in `application/`
6. Adapter implementieren (REST, Persistence)
7. Tests auf allen Ebenen

=== Regeln

* Bei Unklarheiten: *nachfragen*, nicht raten
* Fachbegriffe exakt aus Glossar übernehmen
* Keine Änderungen an der Fachlichkeit ohne Rücksprache
* Commits nach jeder logischen Einheit

== Best Practices

=== Kontext-Management

Das wichtigste Ressource in Claude Code ist das *Kontext-Fenster*. Es füllt sich schnell mit Dateiinhalten und Command-Outputs.

[TIP]
====
* `/compact` bei ~50% Kontextfüllung ausführen
* `/clear` beim Wechsel zu neuen Aufgaben
* Kleine, fokussierte Prompts statt Mega-Anfragen
====

=== Thinking-Modi

Claude Code unterstützt verschiedene Denk-Intensitäten:

[cols="1,2"]
|===
| Keyword | Verwendung

| `think`
| Standard-Überlegung

| `think hard`
| Komplexere Probleme

| `think harder`
| Architektur-Entscheidungen

| `ultrathink`
| Maximale Analyse (z.B. Security-Review)
|===

.Beispiel
[source]
----
Ultrathink: Analysiere die Sicherheit unserer
Authentifizierungs-Implementierung.
----

=== Subagents für spezialisierte Aufgaben

Für komplexe Reviews können dedizierte Subagents definiert werden:

.`.claude/agents/security-reviewer.md`
[source,markdown]
----
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---

Du bist ein Senior Security Engineer. Prüfe Code auf:
- Injection-Schwachstellen (SQL, XSS, Command Injection)
- Authentifizierungs- und Autorisierungsfehler
- Secrets oder Credentials im Code
- Unsichere Datenverarbeitung

Gib konkrete Zeilenreferenzen an.
----
